§ 01Purpose

Offline evaluation of extraction quality against a human-labeled CSV. Produces per-field accuracy / precision / recall / F1, a per-cell comparison table with the model's confidence attached, and a calibration check on whether extract_conf correlates with correctness. Independent of notebook 02 — both can run after notebook 01 in any order.

§ 02Inputs and outputs

Inputs
  • deeds_extracted_flat — predictions, extract_conf, citation_ids.
  • data/GroundTruth.csv in a UC volume — column-oriented (one column per document).
Outputs
  • deeds_ground_truth — transposed GT, one row per document.
  • deeds_extracted_vs_gt — per-cell comparison + high_conf_wrong.
  • deeds_eval_metrics_gt — per-field accuracy / precision / recall / F1.
  • deeds_calibration_gt — conf-bin counts and accuracy.
  • MLflow run under /Users/q.yu@databricks.com/deeds_poc_eval_gt.

§ 03Core methodology

AspectApproach
ComparatorHybrid: fuzzy (Levenshtein ≥ 0.85) for DocumentTitle; strict equality for IDs and dates after normalization.
Absence handlingBlank-when-absent counts as a true negative. Five outcomes: TP, TN, FP, FN, FP_FN.
ArchitectureTable-first. Match logic in PySpark / SQL; MLflow scorers are pass-throughs that surface precomputed classifications with informative rationale.
CalibrationBins extract_conf into four ranges; per-bin accuracy + monotonicity verdict.

§ 04Design rules

  • Null is a meaningful value. GT records blank when the entity isn't in the document. Both comparator and metrics treat (null, null) as a correct true-negative — not a missing answer.
  • Per-field comparators, not one-size-fits-all. Free-form text gets fuzzy; numeric IDs get strict-after-normalization. A single-digit difference in DocumentNumber is a real error and must not be hidden by fuzzy logic.
  • Both-non-null mismatches count twice. FP_FN is penalized in both precision and recall denominators (and 2× in F1's denominator). One wrong prediction blocked the right one.
  • Table-first. deeds_extracted_vs_gt is a deliverable. Match logic in PySpark / SQL; MLflow scorers carry the same data with informative per-row rationale.

§ 05Load and join the ground truth

pd.read_csv with header=None so rows / columns can be addressed by index. Two normalizers keep the join robust:

  • _normalize_image_name — canonical join key: lowercase, replace runs of [\s:_-]+ with _. Resolves CSV / filename mismatches like ARPULA:2023 00023192 → arpula_2023_00023192 matching ARPULA-2023_00023192.TIF.
  • _stringify — preserves leading zeros (e.g. 01161191), treats blanks and NaN as None, and strips trailing .0 from pandas-read floats that were really integers.

Blanks land as null in Delta — never empty strings. Inner join on join_key; the smoke check is row count == number of GT documents.

Per-field columns added to the joined frame:

  • <field>_exact_match — strict equality after normalization.
  • <field>_fuzzy_match — same as exact except DocumentTitle, which uses Levenshtein. The raw similarity is stored as DocumentTitle_levenshtein_similarity.
  • <field>_classification — categorical TP / TN / FP / FN / FP_FN.

§ 07deeds_extracted_vs_gt — long-format comparison

Pivots from one row per document to one row per (image_name, field). Each row carries:

  • gt_value, pred_value, the match flags, levenshtein_similarity (DocumentTitle only), and classification.
  • extract_conf, citation_ids, and cited_regions (JSON-stringified bboxes resolved from the doc's full citations array).
  • high_conf_wrong — defined as classification ∈ {FP, FN, FP_FN} AND extract_conf ≥ 0.8. The headline error signal.

Display ordering: high_conf_wrong DESC, errors before correct, extract_conf DESC. _resolve_citations filters the doc's full citations array to entries whose id is in the cell's citation_ids.

§ 08Per-field corpus metrics

SQL aggregation over the long-format table. Per field:

accuracy  = (tp + tn) / n
precision = tp / (tp + fp + fp_fn)
recall    = tp / (tp + fn + fp_fn)
f1        = 2*tp / (2*tp + fp + fn + 2*fp_fn)

fp_fn appears in both precision and recall denominators because a both-non-null mismatch is simultaneously a hallucination and a miss.

§ 09MLflow scorers and run logging

  • _build_expected_facts — for each field, generates "<field> is <value>" if GT has a value, or "<field> is not present in the document" if null. The negative statement enables Correctness to evaluate legitimate absences.
  • _make_field_scorer — factory producing per-field pass-through scorers. The match logic was computed in §06; the scorer surfaces the classification with a rich rationale string ("classification=TP; comparator=levenshtein_sim≥0.85 on lowercase+alnum+collapse_ws; gt='THIS DEED', pred='THIS DEED', levenshtein_sim=1.000").
  • COMPARATOR_DESCRIPTION — small dict labeling each field's comparator and normalization, used in the rationale strings. Single source of truth for the human-readable description.

Scorers passed to evaluate(): built-in Correctness + six per-field scorers. A second logging step reuses run_id=results.run_id so per-field corpus metrics land alongside the scorer-level metrics. F1 isn't a row-wise mean and must be logged from the SQL aggregation.

§ 10Confidence calibration check

Bins extract_conf into [0, 0.5), [0.5, 0.8), [0.8, 0.95), [0.95, 1.0]. Per bin: count of TP+TN vs FP+FN+FP_FN, accuracy. Null-confidence cells excluded.

CautionWith ≤36 cells, treat the verdict as a sanity signal not a calibrated curve. Watch the trend across batches.

§ 11Reviewer workflow

  1. Open deeds_eval_metrics_gt. Identify which fields are best and worst by F1.
  2. Open deeds_extracted_vs_gt, sort high_conf_wrong DESC then extract_conf DESC. Top rows are the most damaging errors — the model was confident and wrong.
  3. For each row, use the cited_regions JSON to jump to the cited bbox / page in the source document. Compare to gt_value and pred_value.
  4. Open deeds_calibration_gt. Check the monotonicity verdict. If False, treat extract_conf as advisory only.
  5. The MLflow run UI shows scorer metrics, per-field correctness, and calibration metrics in one view — useful for cross-version comparisons when schema or instructions change.

§ 13Failure modes to watch

  • Inner join produces fewer rows than expected. _normalize_image_name doesn't cover a new naming pattern. Print mismatched keys, extend the regex.
  • All FP_FN and no TP. Comparator normalization is too strict. Inspect norm_* outputs on a few examples.
  • Calibration verdict says non-monotonic. Confidence is unreliable on this corpus. Raise HIGH_CONF_THRESHOLD and the production review cutoff in notebook 02.
  • Correctness scorer disagrees with per-field scorers. Judge model is reading the negative expected_facts differently than the strict comparator. Investigate per row; usually the comparator is correct and the judge is lenient.
